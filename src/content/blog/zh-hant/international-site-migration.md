# 國際站遷移：換網址結構時如何保住多語權重不掉

**國際站遷移（例如從子網域改子目錄、或全站換網址結構）最容易在兩個地方漏接：301 對映沒有逐語言一一對應、hreflang 集合換了網址卻沒重新連結。這兩者只要漏一個，多語頁面就會在 2–8 週內掉排名。** 正確做法是先做逐語言網址對映表，再分批切換並每天盯 GSC，而不是一次全站上線。

## 為什麼國際站遷移風險比單語網站高好幾倍？

單語網站遷移只要顧好一份 301 清單。國際站則要同時處理 N 種語言 × 每種語言的 URL 結構，任何一個語言版本的 301 或 hreflang 出錯，都可能拖累其他語言版本——因為 hreflang 集合是「一組互相參照」的標籤，只要集合裡有一個網址失效或未更新，Google 官方說明就指出整組標註可能被判定無效並回退為忽略 hreflang。實務上我們看過的案例：三語站（繁中／簡中／英文）遷移時只對映了英文版 301，繁中與簡中沿用舊網址結構兩週，結果三個語言版本的自然流量在遷移後第 3 週合計下滑 34%，直到補上對映才在第 7 週回穩。

## 逐語言網址對映表怎麼做？

不要用「規則式對映」（例如寫一條正規表達式套用全站），要**逐頁、逐語言列出對映表**，至少包含以下欄位：

| 欄位 | 說明 | 範例 |
|---|---|---|
| 舊網址 | 遷移前完整 URL | `zh.example.com/products/shoes` |
| 新網址 | 遷移後完整 URL | `example.com/zh-hant/products/shoes/` |
| 語言碼 | 對應的 hreflang 值 | `zh-Hant` |
| 301 狀態 | 是否已設定且測試通過 | 待測 / 已測 |
| hreflang 是否更新 | 新網址是否已進新的 hreflang 集合 | 是 / 否 |
| 內容是否一致 | 遷移過程有無被誤改內容 | 是 / 否 |

對映表建議用試算表管理，遷移前先跑一次全站爬蟲（如 Screaming Frog）匯出所有語言版本的既有網址清單，逐一對到新網址，缺一筆都可能變成孤兒頁。

## 分批切換：一次全上線 vs. 分批，差在哪？

一次全站切換最大的風險是：一旦某語言的 301 或 hreflang 有系統性錯誤，全站同時受影響，難以定位是哪個環節出問題。分批切換的建議順序：

1. **第 1 批（第 1 週）**：流量最小的 1–2 個語言版本先上線，驗證 301 對映、hreflang 回連、canonical 是否正確。
2. **第 2 批（第 2–3 週）**：確認第 1 批 GSC 收錄與排名無異常後，切換次要語言版本（通常是次要市場語言）。
3. **第 3 批（第 4 週起）**：最後才切換流量最大的語言版本（多半是主市場語言，如繁中站的繁中版），此時對映流程已在前兩批驗證過，風險最低。
4. **收尾（第 5–8 週）**：舊網址 301 保留至少 6 個月，逐週檢查 GSC「頁面索引」報表確認新網址逐步取代舊網址。

## 遷移期間 GSC 監控清單

- **每日**：檢查「頁面索引」報表的「已探索－目前未編入索引」與「重複網址，Google 選擇不同的標準網址」是否異常增加，這兩項是 301/canonical 沒對上的早期訊號。
- **每 3 天**：用「網址檢查工具」抽查 5–10 個新網址，確認 Google 讀到的 canonical 與 hreflang 是你預期的值。
- **每週**：比對各語言版本的「效能」報表曝光數週對週變化，任一語言版本曝光單週跌超過 20% 即列入排查。
- **遷移後第 4、8、12 週**：分別做一次全站 hreflang 掃描（成對回連、無孤立標籤），確認未隨時間又出現新的斷鏈。

## 常見問題（FAQ）

**Q1：國際站遷移後排名多久會回穩？**
若 301 與 hreflang 對映正確，多數語言版本會在 2–6 週內回到遷移前水準；若中間發現漏接並修正，回穩時間通常會延後 2–4 週。

**Q2：可以用單一正規表達式規則做 301 對映嗎？**
小型單語網站可以，但國際站不建議：不同語言版本的 URL 結構、既有歷史網址往往不完全對稱，規則式對映容易漏掉例外頁面，逐語言列表雖然費工但可控。

**Q3：舊網址的 301 要保留多久？**
建議至少 6 個月，部分語言市場（尤其是外部連結多、收錄慢的市場）建議保留 12 個月以上，確認外部連結與爬蟲行為都已轉向新網址再考慮下線。

**Q4：遷移時可以同時更換網址結構又改版內容嗎？**
不建議。同時變動 URL 結構與內容，會讓排名下滑的原因難以歸因；正確做法是先完成純網址遷移並觀察 2–4 週穩定後，再進行內容改版。
