# 多語網址的 slug 策略：要不要翻譯 slug、要不要用當地語言

**多語 slug 沒有全域最優解：英文 slug（如 `/de/pricing/`）好管理、好對照、不怕特殊字元編碼問題，但當地語言 slug（如 `/de/preise/`）在 SERP 上能讓當地用戶多看一眼、實測 CTR 平均高出 8–15%。經驗法則是「高流量、當地語意義明確的頁面翻 slug；結構性、跨語言一致的頁面（分類、標籤、系統頁）留英文」。**

## 為什麼這是個真問題？

Slug 出現在 URL 列與 SERP 網址預覽上，是使用者點擊前唯一會看到的「非標題文字」。當地語言 slug 讓當地用戶一眼認出「這是我看得懂的頁面」，尤其在非拉丁字母語系（阿拉伯語、日語、韓語）差異更明顯——但翻譯 slug 也帶來三個實際麻煩：URL 編碼後又長又醜（`%E4%B8%AD%E6%96%87` 這類）、跨語言對照與維護成本上升、以及每次改標題是否要跟著改 slug 的兩難。

## 兩種策略的量化比較

| 面向 | 英文 slug（如 `/ja/pricing/`） | 當地語言 slug（如 `/ja/料金/` 或羅馬拼音 `/ja/ryokin/`） |
|---|---|---|
| SERP CTR | 基準值 | 平均高 8–15%（當地用戶辨識度高） |
| URL 可讀性（非拉丁字母） | 全站一致、無編碼問題 | 若用原生文字，URL 顯示為 `%E6%96%99...` 編碼亂碼，多數瀏覽器雖會解碼但分享時常出錯 |
| 跨語言維護成本 | 低：一套 slug 對照表，改版時只認英文 key | 高：N 種語言 = N 套 slug，改標題常連動改 slug，易產生舊 slug 404 |
| 適合頁型 | 系統頁、分類頁、工具頁、Boss 頁 | 高意圖文章頁、在地商業頁、行銷活動頁 |
| SEO 直接影響 | 中性（Google 不因 slug 語言加減分） | 中性偏正（slug 中的關鍵字若與查詢字面相符，SERP 加粗片段增加，間接助攻 CTR） |

## 分語言的具體建議

1. **拉丁字母語系（英、法、德、西、葡）**：直接翻譯 slug，URL 不會出現編碼問題，翻譯成本也低。例：`/fr/tarifs/` 優於 `/fr/pricing/`。
2. **非拉丁字母且有羅馬拼音慣例的語系（日語、韓語）**：用羅馬拼音而非原生文字，兼顧可讀性與網址整潔。例：`/ja/ryokin/` 優於原生日文 slug。
3. **非拉丁字母且無標準羅馬化的語系（阿拉伯語、泰語）**：保留英文 slug，靠 `hreflang` 與頁面內容做語言辨識，不硬翻。
4. **中文（繁體／簡體）**：英文 slug 是本站與多數繁簡雙語站的實務選擇——中文 slug 編碼後在分享連結、Threads/社群貼文中容易被截斷或顯示亂碼，英文 slug 更穩定。
5. **系統性頁面（分類、標籤、篩選器、Boss 頁）不論語言一律用英文 slug**，換取跨語言維護時「一套 slug 對照表走天下」的效率；只有面向終端讀者、承載大量自然搜尋流量的內容頁，才值得為 CTR 投入翻譯 slug 的維護成本。

## 常見問題（FAQ）

**Q1：改 slug 語言會不會影響既有排名？**
會有短期波動。改 slug 等同換 URL，必須設 301 轉址、更新 hreflang 互連、並在 GSC 提交新網址索引，通常 2–6 週排名回穩，期間避免同時大改內容，以免訊號混淆。

**Q2：slug 該用原生文字還是羅馬拼音？**
沒有絕對答案，取決於語系。有成熟羅馬化慣例的語言（日語、韓語）建議用拼音；沒有的語言（中文）建議直接用英文 slug，避免 URL 編碼問題。

**Q3：slug 長度有沒有建議上限？**
建議 3–5 個字（詞）以內，涵蓋目標關鍵字即可，避免把整個標題塞進 slug；過長 slug 在 SERP 截斷、在社群分享時也不美觀。

**Q4：多語 slug 一定要跟 hreflang 綁在一起考慮嗎？**
是的。無論 slug 翻不翻譯，每個語言版本都要有正確的自我參照 hreflang 與雙向回連，否則就算 slug 選對了，Google 仍可能誤判各語言版為重複內容。
