# 新聞與媒體站的 GEO：時效內容如何被 AI 即時引用

**新聞站的 GEO 和長青文完全是兩套邏輯：長青文靠被 AI「長期記住」，新聞靠被 AI「即時抓到」。當 ChatGPT、Perplexity、Google AI Overview 回答一則正在發生的事件時，它們是現場去檢索最新網頁，而不是翻模型裡的舊記憶——所以能不能進即時答案，取決於三個機器可讀的訊號：清楚的時間戳、事件實體標註、穩定的 URL。** 這三件事缺一項，你的獨家報導都可能被 AI 略過，改引一家把訊號做齊的競業。

## 為什麼新聞 GEO 不能照長青文那套做？

因為 AI 面對時效查詢時，走的是「檢索」路徑而非「記憶」路徑。使用者問「某公司今天發生什麼事」，模型的訓練資料早已過期，於是引擎會即時發爬蟲抓最新結果、重排、再生成帶來源的答案。這條即時管線在意的是**新鮮度與可信度訊號**，而長青文優化的網域權重、內部連結網絡在這裡幫助有限——一篇三小時前發布、訊號齊全的報導，會贏過一篇排名很高但沒有明確時間戳的舊頁。所以新聞站要贏的不是「排名」，是「被即時管線判定為此刻最相關、最可信的來源」。

## 三要件之一：清楚且機器可讀的時間戳

AI 判斷「這是不是最新的」，第一眼看的不是內文寫的日期，是結構化的時間欄位。實務上要做齊：

- **`datePublished` 與 `dateModified`** 同時放進 NewsArticle schema，格式用 ISO 8601（含時區，如 `2026-08-05T09:30:00+08:00`）。
- **可見的發布時間**與 schema 一致，別頁面寫「今天上午」但 schema 是三天前。
- **重大更新就更新 `dateModified`**，並在文中標注「更新於 …」——即時查詢中，較新的 `dateModified` 常是被選中的關鍵。
- **別用相對時間當唯一訊號**：「2 小時前」對人友善，但機器要的是絕對時間戳。

一句話：時間戳是新聞 GEO 的第一張門票，模糊或矛盾的日期等於自我降權。

## 三要件之二：事件與實體標註

即時答案是「圍繞某個事件或實體」組織的。AI 要先確認「這篇講的正是使用者問的那個事件、那家公司、那個人」，才敢引用。強化實體消歧的做法：

1. **NewsArticle schema 補齊 `about` 與 `mentions`**，用實體 URL（維基百科、官方網站、`sameAs`）指明本文談的主體，而非只留純文字。
2. **標題與首段就點名核心實體全稱**，別只用代稱或縮寫，讓機器一眼綁定。
3. **重大事件加上背景區塊**，把「這是什麼事件、涉及哪些實體」講清楚，方便 AI 把你的報導掛到正確的事件節點上。
4. **標注地點與組織實體**（`Place`、`Organization`），在地或產業事件尤其吃這個。

實體標註做得好，等於幫 AI 把你的報導精準歸檔到它正在回答的那個問題底下。

## 三要件之三：穩定且乾淨的 URL

即時管線抓到頁面的那一刻，URL 必須能穩定、快速地回傳正文——這是最常被忽略、卻最致命的一環：

- **URL 上線後不要改**：改網址又沒做 301，等於把剛被 AI 記下的來源變成死連結。
- **別把正文鎖在 JS 後面**：若初始 HTML 不含內文，多數 AI 爬蟲讀到的是空殼，抓不到就不會引用（技術細節見〈[AI 爬蟲清單與管理](/zh-hant/blog/ai-crawler-management/)〉）。
- **別擋即時檢索爬蟲**：OAI-SearchBot、PerplexityBot 等「檢索型」爬蟲若被 robots.txt 或 CDN 擋掉，你就直接退出即時答案的候選池。
- **快、不要牆**：即時抓取講求速度，付費牆或反爬蟲驗證會讓引擎放棄你這一則。

## 新聞 GEO vs 長青文 GEO：一張表看差異

| 面向 | 新聞／時效內容 | 長青內容 |
|---|---|---|
| AI 取材路徑 | 即時檢索（現抓現用） | 長期收錄＋重排 |
| 最關鍵訊號 | 時間戳、實體、穩定 URL | 網域權重、內部連結、內容深度 |
| 新鮮度重要性 | 極高（小時級決勝） | 中等（季更即可） |
| 首段該寫什麼 | 事件事實：誰、何時、發生什麼 | 自足的定義或結論 |
| 主要 schema | NewsArticle（含 date、about） | Article、FAQPage |
| 成敗判斷 | 有沒有進「此刻」的答案 | 有沒有被長期引用 |

兩者不衝突：新聞站也該有長青的背景解釋頁承接搜尋，但時效報導本身，優化重點必須切到即時管線這一側。內容結構的通用原則，可參〈[GEO 內容結構](/zh-hant/blog/geo-content-structure/)〉。

## 落地檢查清單（發布前 60 秒）

1. NewsArticle schema 有 `datePublished`＋`dateModified`（ISO 8601 含時區）。
2. 頁面可見時間與 schema 一致，非只寫相對時間。
3. `about`／`mentions` 用實體 URL 標明核心事件與主體。
4. 標題與首段點名核心實體全稱。
5. 初始 HTML 已含完整正文（非 JS 才載入）。
6. OAI-SearchBot、PerplexityBot、Google-Extended 未被 robots.txt／CDN 誤擋。
7. URL 為永久網址，上線後不再變動。

## 常見問題（FAQ）

**Q1：新聞站到底要不要做傳統 SEO？**
要，但要分工。傳統 SEO 幫你的長青背景頁長期排名；即時報導則要另外顧好時間戳、實體與爬蟲存取這條即時管線。兩者是疊加，不是二選一。

**Q2：只要發得夠快，就會被 AI 引用嗎？**
不夠。速度是必要條件不是充分條件。發得快但沒有機器可讀的時間戳、正文鎖在 JS 後、或檢索爬蟲被擋，AI 一樣抓不到、引不了。三要件要一起到位。

**Q3：一定要用 NewsArticle schema 嗎？用 Article 可以嗎？**
時效內容建議用 NewsArticle，它的語意明確告訴引擎「這是有時效的新聞」，並支援日期與事件實體欄位。Article 可用但語意較弱，時間與事件訊號的表達不如 NewsArticle 精準。

**Q4：怎麼知道自己的報導有沒有進即時答案？**
在事件熱度期間，拿相關問題去問 ChatGPT（開搜尋）、Perplexity、Google AI Overview，看來源清單有沒有你。系統化的追蹤方法，見〈[AI 搜尋可見度怎麼追蹤](/zh-hant/blog/track-ai-visibility/)〉。

**Q5：付費牆的新聞內容還有機會被引用嗎？**
會大打折扣。即時爬蟲讀不到牆後正文就無從引用。折衷做法是提供夠完整的免費摘要區塊（含核心事實與 schema），讓 AI 至少抓得到可引用的段落，再引導點擊看全文。
