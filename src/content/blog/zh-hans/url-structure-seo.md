# URL 结构与网站架构怎么规划

**URL 结构的四个原则：短而可读、用语义化目录分层（如 /blog/、/services/）、slug 用英文小写连字符、一旦发布永不更改。网站架构则遵循「任何页面从首页三次点击内可达」，并让目录层级对应主题群集。** 架构是发布前的决策——上线后再改，每次都是一场搬家。

## 为什么 URL 结构值得在第一天想清楚？

1. **Google 用目录理解网站区块**：清楚的路径（`/zh-hans/blog/...`）让爬虫与 GSC 报表都能按文件夹分析。
2. **改 URL 的代价极高**：每次更改都需要 301 重定向，权重传递有折损风险，外部既有链接全部要靠重定向接——所以第一原则是「设计成永远不用改」。
3. **用户与 AI 都在读 URL**：`example.com/blog/what-is-content-debt` 在搜索结果、社群分享、AI 引用清单里都比 `example.com/p?id=8471` 可信。

## URL 命名七原则

1. **小写英文＋连字符**：`what-is-content-debt`。Google 官方建议用连字符（-）不用下划线（_）；大小写混用会制造重复网址。
2. **短，砍掉赘字**：slug 放核心语义即可，3–6 个英文单词是舒适区。`/blog/internal-linking-strategy/` 而非 `/blog/how-to-build-an-internal-linking-strategy-for-your-website/`。
3. **不放日期**：`/2026/08/17/` 让内容看起来会过期，而且常青文更新后 URL 上的旧日期变成谎言。
4. **不放会变动的分类**：文章换分类不该换 URL——博客文章统一放 `/blog/<slug>/`，分类用标签或 frontmatter 管理，不进路径。
5. **中文站 slug 建议用英文**：中文 URL 分享时会变成一长串百分比编码（`%E5%85%A7%E5%AE%B9...`），丑且容易断裂。中文关键词放 title 与正文就够。
6. **避免参数当内容路径**：`?id=`、`?cat=` 这类参数页难管理、易重复；参数留给筛选与追踪，并用 canonical 收敛。
7. **尾斜线统一**：`/blog/foo` 和 `/blog/foo/` 选一种，另一种 301 过去——否则就是全站规模的重复网址。

## 网站架构：目录对应主题群集

以本站这种「博客＋服务页」的 SEO/GEO 网站为例：

```
/zh-hans/                      ← 语言根
├── blog/<slug>/               ← 所有文章（长尾文＋支柱页）
├── geo/                       ← Boss 页：GEO 服务＋工具
├── seo-audit/                 ← Boss 页：SEO 体检服务
└── about/                     ← 信任页
```

要点：

- **层级浅**：内容页最多两层目录。Google 官方说明 URL 深度本身不是排名因素，但每多一层，内部链接维护与爬取路径就多一分复杂。
- **架构的「深度」由内部链接决定，不是由目录决定**：一页就算路径只有一层，若从首页要点五次才到，它就是深页。目标：**任何重要页面 ≤ 3 次点击可达**。灌权重的链接设计是另一个专题（内部链接策略）。
- **主题群集靠链接成形，不必靠子目录**：同群集文章统一放 `/blog/`、用支柱页互链串起来即可；为每个群集开子目录（`/blog/technical-seo/...`）也可以，但代价是文章换群集就得搬家。

## 扁平 vs 深层：怎么选？

| 架构 | 适合 | 风险 |
|---|---|---|
| 扁平（所有文章在 /blog/ 下） | 博客、内容站（≤ 数千页） | 目录报表无法按主题切分（可用 GSC 筛选字符串补救） |
| 深层（依主题开子目录） | 电商、大型知识库（数万页+） | 分类调整成本高、容易产生重复路径 |

多数内容网站选扁平就对了——简单、稳定、永不搬家。

## 多语言站的路径设计

用**语言子目录**是多数情况的最佳解：`/zh-hant/`、`/zh-hans/`、`/en/`，配合 hreflang 标记。相比子域名（`en.example.com`）或 ccTLD（`example.tw`），子目录让所有语言共享同一域名权重，维护成本也最低。注意各语言的 slug 各自独立规划，不要用机器直译的路径。

## 已经上线的站想改 URL，值得吗？

先问：现在的结构有造成实际伤害吗（重复网址、参数地狱、报表无法分析）？

- **没有** → 不要改。丑但无害的 URL 不值得一次搬家。
- **有** → 集中一次改到位，准备完整的一对一 301 对照表，一次搬完，别分批慢慢改。

## 常见问题（FAQ）

**Q1：URL 里放关键词对排名有帮助吗？**
有，但非常轻微——Google 官方说它是「很小的排名因素」。放的理由主要是可读性与点击信任感，不要为了塞关键词把 slug 拉长。

**Q2：.html 结尾需要拿掉吗？**
不影响排名。新站建议用干净路径（无扩展名）纯粹为了美观与一致；旧站有 .html 就留着，不值得为此重定向。

**Q3：文章 slug 发布后发现打错字，改不改？**
刚发布（尚未收录、无外链）就直接改；已收录就权衡——错字不影响功能的话留着，要改就必须设 301。

**Q4：URL 深度会影响 crawl budget 吗？**
间接会：深层页若内部链接也少，爬取优先度就低，容易卡在「Discovered - currently not indexed」。解法是链接，不是硬把目录砍层——这是需求端的问题。
