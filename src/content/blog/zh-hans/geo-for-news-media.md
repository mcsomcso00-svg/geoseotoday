# 新闻与媒体站的 GEO：时效内容如何被 AI 即时引用

**新闻站的 GEO 和常青文完全是两套逻辑：常青文靠被 AI「长期记住」，新闻靠被 AI「即时抓到」。当 ChatGPT、Perplexity、Google AI Overview 回答一则正在发生的事件时，它们是现场去检索最新网页，而不是翻模型里的旧记忆——所以能不能进即时答案，取决于三个机器可读的信号：清楚的时间戳、事件实体标注、稳定的 URL。** 这三件事缺一项，你的独家报道都可能被 AI 略过，改引一家把信号做齐的同业。

## 为什么新闻 GEO 不能照常青文那套做？

因为 AI 面对时效查询时，走的是「检索」路径而非「记忆」路径。用户问「某公司今天发生什么事」，模型的训练数据早已过期，于是引擎会即时派爬虫抓最新结果、重排、再生成带来源的答案。这条即时管线在意的是**新鲜度与可信度信号**，而常青文优化的域名权重、内部链接网络在这里帮助有限——一篇三小时前发布、信号齐全的报道，会赢过一篇排名很高但没有明确时间戳的旧页。所以新闻站要赢的不是「排名」，是「被即时管线判定为此刻最相关、最可信的来源」。

## 三要件之一：清楚且机器可读的时间戳

AI 判断「这是不是最新的」，第一眼看的不是正文写的日期，是结构化的时间字段。实务上要做齐：

- **`datePublished` 与 `dateModified`** 同时放进 NewsArticle schema，格式用 ISO 8601（含时区，如 `2026-08-05T09:30:00+08:00`）。
- **可见的发布时间**与 schema 一致，别页面写「今天上午」但 schema 是三天前。
- **重大更新就更新 `dateModified`**，并在文中标注「更新于 …」——即时查询中，较新的 `dateModified` 常是被选中的关键。
- **别用相对时间当唯一信号**：「2 小时前」对人友好，但机器要的是绝对时间戳。

一句话：时间戳是新闻 GEO 的第一张门票，模糊或矛盾的日期等于自我降权。

## 三要件之二：事件与实体标注

即时答案是「围绕某个事件或实体」组织的。AI 要先确认「这篇讲的正是用户问的那个事件、那家公司、那个人」，才敢引用。强化实体消歧的做法：

1. **NewsArticle schema 补齐 `about` 与 `mentions`**，用实体 URL（维基百科、官方网站、`sameAs`）指明本文谈的主体，而非只留纯文本。
2. **标题与首段就点名核心实体全称**，别只用代称或缩写，让机器一眼绑定。
3. **重大事件加上背景区块**，把「这是什么事件、涉及哪些实体」讲清楚，方便 AI 把你的报道挂到正确的事件节点上。
4. **标注地点与组织实体**（`Place`、`Organization`），本地或行业事件尤其吃这个。

实体标注做得好，等于帮 AI 把你的报道精准归档到它正在回答的那个问题底下。

## 三要件之三：稳定且干净的 URL

即时管线抓到页面的那一刻，URL 必须能稳定、快速地返回正文——这是最常被忽略、却最致命的一环：

- **URL 上线后不要改**：改网址又没做 301，等于把刚被 AI 记下的来源变成死链接。
- **别把正文锁在 JS 后面**：若初始 HTML 不含正文，多数 AI 爬虫读到的是空壳，抓不到就不会引用（技术细节见〈[AI 爬虫清单与管理](/zh-hans/blog/ai-crawler-management/)〉）。
- **别挡即时检索爬虫**：OAI-SearchBot、PerplexityBot 等「检索型」爬虫若被 robots.txt 或 CDN 挡掉，你就直接退出即时答案的候选池。
- **快、不要墙**：即时抓取讲求速度，付费墙或反爬虫验证会让引擎放弃你这一则。

## 新闻 GEO vs 常青文 GEO：一张表看差异

| 面向 | 新闻／时效内容 | 常青内容 |
|---|---|---|
| AI 取材路径 | 即时检索（现抓现用） | 长期收录＋重排 |
| 最关键信号 | 时间戳、实体、稳定 URL | 域名权重、内部链接、内容深度 |
| 新鲜度重要性 | 极高（小时级决胜） | 中等（季更即可） |
| 首段该写什么 | 事件事实：谁、何时、发生什么 | 自足的定义或结论 |
| 主要 schema | NewsArticle（含 date、about） | Article、FAQPage |
| 成败判断 | 有没有进「此刻」的答案 | 有没有被长期引用 |

两者不冲突：新闻站也该有常青的背景解释页承接搜索，但时效报道本身，优化重点必须切到即时管线这一侧。内容结构的通用原则，可参〈[GEO 内容结构](/zh-hans/blog/geo-content-structure/)〉。

## 落地检查清单（发布前 60 秒）

1. NewsArticle schema 有 `datePublished`＋`dateModified`（ISO 8601 含时区）。
2. 页面可见时间与 schema 一致，非只写相对时间。
3. `about`／`mentions` 用实体 URL 标明核心事件与主体。
4. 标题与首段点名核心实体全称。
5. 初始 HTML 已含完整正文（非 JS 才加载）。
6. OAI-SearchBot、PerplexityBot、Google-Extended 未被 robots.txt／CDN 误挡。
7. URL 为永久网址，上线后不再变动。

## 常见问题（FAQ）

**Q1：新闻站到底要不要做传统 SEO？**
要，但要分工。传统 SEO 帮你的常青背景页长期排名；即时报道则要另外顾好时间戳、实体与爬虫存取这条即时管线。两者是叠加，不是二选一。

**Q2：只要发得够快，就会被 AI 引用吗？**
不够。速度是必要条件不是充分条件。发得快但没有机器可读的时间戳、正文锁在 JS 后、或检索爬虫被挡，AI 一样抓不到、引不了。三要件要一起到位。

**Q3：一定要用 NewsArticle schema 吗？用 Article 可以吗？**
时效内容建议用 NewsArticle，它的语义明确告诉引擎「这是有时效的新闻」，并支持日期与事件实体字段。Article 可用但语义较弱，时间与事件信号的表达不如 NewsArticle 精准。

**Q4：怎么知道自己的报道有没有进即时答案？**
在事件热度期间，拿相关问题去问 ChatGPT（开搜索）、Perplexity、Google AI Overview，看来源清单有没有你。系统化的追踪方法，见〈[AI 搜索可见度怎么追踪](/zh-hans/blog/track-ai-visibility/)〉。

**Q5：付费墙的新闻内容还有机会被引用吗？**
会大打折扣。即时爬虫读不到墙后正文就无从引用。折衷做法是提供够完整的免费摘要区块（含核心事实与 schema），让 AI 至少抓得到可引用的段落，再引导点击看全文。
